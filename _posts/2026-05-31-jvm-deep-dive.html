Part 1

What Is the JVM?

The runtime that makes Java possible — and why it's more interesting than you think

Most developers know "the JVM runs Java code" and move on. But inside that sentence is a system that manages millions of live objects, compiles code on the fly, and reclaims memory — all while your application serves requests without pausing.

When you run javac, you don't get CPU-specific machine code. You get bytecode — a platform-neutral instruction set stored in .class files. The JVM reads bytecode and translates it to native instructions for whatever hardware it's running on.

🏭 Production Reality — JRE vs JDK in Containers

In Docker-based Kubernetes deployments, ship the JRE — not the full JDK — to keep image sizes small. But when an incident happens and you need jstack, jmap, or jfr, those tools are only in the JDK. Solution: run a debug sidecar container with JDK tools that can nsenter into the app pod. Plan for production diagnostics before you need them at 3 AM.

Part 2

JVM Architecture

Three subsystems. One very smart runtime.

JVM Architecture — Annotated

2.1 The Class Loader — Your App's First Gatekeeper

Before any code runs, the JVM must find and load your .class files. The class loader does this in three phases: Loading (read the bytes), Linking (verify, prepare, resolve), and Initialisation (run static blocks). It uses a parent-delegation model — before loading any class, it asks its parent first, all the way up to the Bootstrap loader. This is why you can't accidentally replace java.lang.String.

🔍 Classloader Leak — Real Incident

A long-running service started hitting OutOfMemoryError: Metaspace. The heap dump showed thousands of ClassLoader instances still alive. The cause: a custom URLClassLoader was being created on every API request to load a plugin JAR, but was never closed. Each loader held class metadata in Metaspace indefinitely. Fix: close the classloader. Lesson — classloaders are resources. Treat them like InputStreams.

2.3 JIT Compilation — Why Java Gets Fast

The JVM starts by interpreting bytecode — simple, but slow for hot paths. As it monitors execution, it identifies "hot methods" (called frequently). These get compiled to native machine code by the JIT. After that, the native code runs directly — no interpretation overhead.

JIT Compilation Tiers — Method Performance Over Time
🏭 JIT Warmup in Kubernetes — Real Fix

After deploying a new service, the first batch of requests were 3–5x slower than steady state. The JIT was still interpreting. Fix: added a /warmup endpoint that fires synthetic requests through all critical code paths. Called from the Kubernetes postStart lifecycle hook before the pod enters the load balancer. First real user request now gets fully JIT-compiled performance.

Garbage Collection

The art of knowing when to let go — step by step, animated

GC is built on a fundamental observation: most objects die young. A web request creates hundreds of temporary objects — nearly all are dead within milliseconds. Only a few live long: caches, connection pools, application context. GC is designed to exploit this fact.

🏭 G1 → ZGC Migration — Real Numbers

A latency-critical API had a strict 200ms p99 SLA. With G1, occasional 180–220ms GC pauses violated the SLA 3–5 times per hour. After switching to ZGC on Java 17, GC pauses dropped to 2–4ms. p99 latency fell from 195ms to 140ms. CPU usage went up ~8% due to concurrent GC work — a completely acceptable tradeoff for a latency-sensitive service.

Part 4

Java Versions — 8 to 21

What changed in each release, and what it meant in real production systems

LTS
Java 8 — Lambdas, Streams, and the Death of PermGen
Released 2014 · Still the most-deployed Java version globally

Java 8 was a paradigm shift. Functional programming arrived. The modern Spring Boot ecosystem was built on it.

  • Lambda expressions — anonymous functions without the anonymous class ceremony
  • Stream API — composable functional pipelines: filter, map, reduce, collect
  • Optional<T> — explicit null handling — no more silent NPE landmines
  • PermGen removed → Metaspace — dynamic native memory, no more PermGen OOM
  • New Date/Time API — finally replaced the broken Calendar class
Java 8
// Before Java 8 — verbose anonymous class + loop
List<String> result = new ArrayList<>();
for (String s : items) {
    if (s.startsWith("active_")) result.add(s.toUpperCase());
}

// Java 8 — streams + lambdas: readable, composable, parallelisable
List<String> result = items.stream()
    .filter(s -> s.startsWith("active_"))
    .map(String::toUpperCase)
    .sorted()
    .collect(Collectors.toList());

// Optional — chain without null checks
String city = user
    .map(User::getAddress)
    .map(Address::getCity)
    .orElse("Unknown");
💥 PermGen was a real pain

Before Java 8, long-running apps that dynamically loaded classes — application servers, hot-reload setups, OSGi — would eventually hit OutOfMemoryError: PermGen space. The fix was always "restart the server." Java 8 moved class metadata to Metaspace in native memory, which grows dynamically. Migrating a legacy Java 7 app to Java 8 and removing PermGen flags eliminated mysterious nightly restarts permanently.

LTS
Java 11 — HTTP Client, Free JFR, and ZGC Arrives
Released 2018 · Enterprise standard for most teams
  • Modern HTTP Client — async, HTTP/2 support, replaces the ancient HttpURLConnection
  • Java Flight Recorder (JFR) free — was Oracle-commercial, now open to everyone
  • String improvements — isBlank(), strip(), lines(), repeat()
  • ZGC experimental — the sub-millisecond GC future begins
Java 11
// HTTP Client — clean, async, HTTP/2
HttpClient client = HttpClient.newBuilder()
    .connectTimeout(Duration.ofSeconds(10))
    .build();
HttpResponse<String> resp = client.send(
    HttpRequest.newBuilder()
        .uri(URI.create("https://api.example.com/data"))
        .GET().build(),
    HttpResponse.BodyHandlers.ofString()
);

// New String methods — genuinely useful
"  hello  ".strip();          // "hello" (unicode-aware, better than trim())
"line1\nline2".lines().count(); // 2
"ha".repeat(3);               // "hahaha"
"   ".isBlank();              // true
🏭 JFR Saves the Day — Latency Spike Investigation

A production service had intermittent latency spikes every few hours with no obvious cause. JFR was enabled with a rolling 5-minute buffer (less than 1% overhead). The next spike triggered a dump. Loaded in JDK Mission Control: the hotspot was immediately visible — a third-party HTTP library using synchronized connection pooling, causing thread contention under load. Found in 10 minutes. Without JFR, this would have taken days.

LTS
Java 17 — Sealed Classes, Records, and Modern Pattern Matching
Released 2021 · Spring Boot 3.x minimum — recommended baseline today
  • Sealed classes — declare exactly which classes can implement a type; compiler enforces exhaustiveness
  • Records — immutable data classes in one line (replaces 50 lines of boilerplate POJO)
  • Switch expressions — switch that returns a value with arrow syntax
  • Helpful NullPointerExceptions — the JVM now tells you exactly which variable was null
  • Pattern matching instanceof — no redundant cast after type check
Java 17
// Records — replaces entire POJO class
record ValidationResult(String id, String status, LocalDateTime at) {}

// Sealed classes — restrict which types exist
sealed interface Result permits Pass, Fail, Pending {}
record Pass(String id, LocalDate validUntil) implements Result {}
record Fail(String id, List<String> faults) implements Result {}
record Pending(String id, String reason) implements Result {}

// Switch expression — exhaustive, returns value, compiler catches missed cases
String action = switch (result) {
    case Pass p    -> "Certified until " + p.validUntil();
    case Fail f    -> "Faults: " + f.faults().size();
    case Pending r -> "Waiting: " + r.reason();
}; // No default needed — compiler knows all cases are covered

// Helpful NPE — before Java 14: just line number
// After: "Cannot invoke 'Address.getCity()' because 'user.getAddress()' is null"
🏭 Sealed Classes Eliminating a Class of Bugs

A workflow platform had a state machine: SUBMITTED → IN_PROGRESS → BLOCKED → COMPLETED / REJECTED. Using plain enums and if-instanceof chains, 3 production incidents in 18 months were caused by code paths that forgot to handle the BLOCKED state. After moving to sealed classes in Java 17, the compiler enforces exhaustiveness at every switch. That entire category of runtime bugs became a compile-time error.

LTS
Java 21 — Virtual Threads: The Biggest JVM Change in a Decade
Released 2023 · Project Loom ships · Millions of concurrent tasks

Virtual Threads are the most impactful change to the JVM since Java 8. They solve the fundamental mismatch between the thread-per-request model and I/O-heavy workloads — without requiring reactive programming.

Traditional Threads vs Virtual Threads — under I/O load
Java 21
// Spring Boot 3.2+ — one property to enable virtual threads
// spring.threads.virtual.enabled=true
// That's it. All @RestController methods run on virtual threads.

// Or explicitly — works just like a regular executor
try (ExecutorService vt = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 10_000).forEach(i ->
        vt.submit(() -> {
            String result = database.query("SELECT ..."); // blocks VT, not OS thread
            cache.set(key, result);
            return result;
        })
    );
} // 10,000 concurrent DB queries — no reactive code, no callbacks

// Java 21 also: Pattern Matching for Switch (standard)
String describe = switch (obj) {
    case Integer i -> "int: " + i;
    case String s  -> "str: " + s.length();
    case null      -> "null";
    default        -> "other";
};
🏭 7x Throughput — One Config Line

A vehicle validation API makes 3–5 downstream calls per request (DB, rules engine, certificate service, audit log). Each request blocked a thread for ~80ms. With a 200-thread pool: ~2,500 req/s before queuing. After spring.threads.virtual.enabled=true on Java 21: 18,000 req/s. Same latency per request. No reactive programming rewrite. No callback hell. One property.

Reading a Thread Dump

🧊 The Frozen Service — Deadlock Diagnosed in 30 Seconds

A microservice stopped responding every few days. Memory fine. CPU near zero. No errors in logs. A thread dump (jstack <pid>) showed it immediately: Thread A holding lock X, waiting for lock Y. Thread B holding lock Y, waiting for lock X. Classic deadlock. JStack even printed "Found one Java-level deadlock." Fix: always acquire locks in the same order. Switched to ConcurrentHashMap and ReentrantLock.tryLock(). Never happened again.

🏭 The 3:14 AM Fix — Full Timeline

T+0: OOM alert fires. Heap dump captured: jcmd 1 GC.heap_dump /tmp/heap.hprof.

T+8min: Eclipse MAT Leak Suspects report: one LinkedHashMap holds 847MB (71% of heap). Traced to a validation rules cache — accumulating unique request keys with no TTL or eviction.

T+31min: Fix: Caffeine cache with expireAfterWrite(10, MINUTES), max 500 entries. Tested locally.

T+47min: Deployed via GitOps rolling restart. Memory stable. Alert resolved. Post-incident: added heap 80% alert to Grafana. Never hit OOM again.
