PHASE 01 Getting Started with SpecKit Install · Initialise · Connect to your microservices
Introduction 01

What is SpecKit?

SpecKit is a specification-first development framework for microservices teams. It treats your API contracts, data schemas, and event definitions as the single source of truth — then gives every role on your team (developers, QA, architects, analysts) AI-augmented tools to work from those specs daily.

In a traditional microservices project, specs drift. Developers implement one thing, the spec says another, QA tests a third version, and the architect's diagrams are six months old. SpecKit solves this by making specs executable, verifiable, and AI-queryable.

Why SpecKit for Microservices?

💡
Core idea: Every microservice publishes a machine-readable spec (OpenAPI for REST, AsyncAPI for events, Avro/Protobuf for schemas). SpecKit indexes all of them into a unified context that AI agents can query, validate against, and generate code from.
ProblemSpecKit Solution

New devs take weeks to understand the systemAI context built from all specs — ask questions in plain English
QA writes tests late in the cycleTest cases generated from specs at design time
ABAs write requirements that devs ignoreRequirements map directly to spec fields and validation rules
Architects lose visibility as the system growsLive dependency graph generated from service specs

Installation 02

Install SpecKit

SpecKit provides a CLI tool and a VS Code extension. Install both to get the full experience.

Prerequisites

  • Node.js 18+ or Java 17+ (CLI available for both runtimes)
  • Git repository with at least one microservice
  • OpenAPI 3.x or AsyncAPI 2.x spec file(s)
  • A Claude API key (for AI-augmented features)

Option A — Install via npm (Recommended)

bash
# Install SpecKit CLI globally
npm install -g @speckit/cli

# Verify installation
speckit --version
# → SpecKit CLI v2.4.1

# Install VS Code extension (optional but recommended)
code --install-extension speckit.speckit-vscode

Option B — Install via Homebrew (macOS/Linux)

bash
brew tap speckit/tap
brew install speckit

Option C — Install via Maven Plugin (Spring Boot projects)

pom.xml
<plugin>
  <groupId>io.speckit</groupId>
  <artifactId>speckit-maven-plugin</artifactId>
  <version>2.4.1</version>
  <executions>
    <execution>
      <goals>
        <goal>validate</goal>
        <goal>generate</goal>
      </goals>
    </execution>
  </executions>
</plugin>

Setup 03

Setup SpecKit for Your Microservices Project

Initialise SpecKit at the monorepo root (or per-service), connect your spec files, and configure team roles.

1

Initialise SpecKit at the project root

Run speckit init in the root of your monorepo or the parent directory containing your microservices.

bash
# Navigate to your project root
cd ~/projects/my-microservices

# Initialise SpecKit (interactive wizard)
speckit init

# → SpecKit will ask:
#   Project name: my-microservices
#   Spec format: OpenAPI / AsyncAPI / Mixed
#   Services to register: (auto-detected)
#   AI provider: Claude / OpenAI / Local
2

Review the generated speckit.config.json

SpecKit creates a config file at the root. Edit it to register all your services and their spec file locations.

speckit.config.json
{
  "project": "my-microservices",
  "version": "1.0.0",
  "ai": {
    "provider": "claude",
    "model": "claude-sonnet-4-6"
  },
  "services": [
    {
      "name": "order-service",
      "path": "./order-service",
      "spec": "./order-service/src/main/resources/openapi.yaml",
      "type": "rest"
    },
    {
      "name": "payment-service",
      "path": "./payment-service",
      "spec": "./payment-service/src/main/resources/openapi.yaml",
      "type": "rest"
    },
    {
      "name": "notification-service",
      "path": "./notification-service",
      "spec": "./notification-service/src/main/resources/asyncapi.yaml",
      "type": "event"
    }
  ],
  "context": {
    "outputDir": ".speckit/context",
    "includeSourceCode": true,
    "includeTests": true
  }
}
3

Set your AI API key

bash
# Set Claude API key (add to ~/.zshrc or ~/.bashrc for persistence)
export ANTHROPIC_API_KEY="sk-ant-..."

# Or store it in SpecKit's secure config
speckit config set ai.apiKey "sk-ant-..."
4

Register all services and verify

bash
# Register all services defined in config
speckit register --all

# Validate all registered specs
speckit validate --all

# List registered services
speckit services list
# → ✓ order-service        (REST)   openapi.yaml v1.2.0
# → ✓ payment-service      (REST)   openapi.yaml v1.0.3
# → ✓ notification-service (EVENT)  asyncapi.yaml v1.1.0

Context Creation 04

Create Context from Source Code Specs

The SpecKit context is a rich, queryable snapshot of your entire microservices landscape — built by combining your spec files with actual source code.

The context is what makes AI features work accurately. Instead of sending raw source code to the AI (expensive and imprecise), SpecKit extracts structured metadata from your specs and code, builds a compact context file, and uses that for all AI queries.

Build the Context

bash
# Build full context for all services
speckit context build --all

# Build context for a single service
speckit context build --service order-service

# Build and include source code analysis
speckit context build --all --include-source --include-tests

# Watch mode: auto-rebuild context when specs change
speckit context watch

What Gets Indexed into the Context?

SourceWhat SpecKit ExtractsUsed By
openapi.yamlEndpoints, request/response schemas, error codes, auth methodsAll roles
asyncapi.yamlTopics/queues, message schemas, producers/consumersDev Arch
Java sourceController classes, service interfaces, DTO fields, annotationsDev QA
Test filesExisting test coverage, test data patternsQA
Git historyRecent changes per service, change frequencyArch QA
Pom.xml / package.jsonDependencies, versions, frameworksArch

Query the Context

bash
# Ask a question about your system in plain English
speckit ask "Which services expose a POST /orders endpoint?"

# Find all services that consume OrderPlaced events
speckit ask "What consumes the OrderPlaced event?"

# Check for schema mismatches between services
speckit context diff order-service payment-service

# Export context for use in custom AI prompts
speckit context export --format markdown --output context.md

PHASE 02 Core SpecKit Workflow Contracts · Code Gen · Validation · Prompts
Contracts 05

Define Service Contracts

SpecKit uses OpenAPI 3.x for REST contracts and AsyncAPI 2.x for event-driven contracts. Here's how to write them effectively for microservices.

REST Contract — order-service

openapi.yaml
openapi: "3.0.3"
info:
  title: Order Service API
  version: "1.2.0"
  description: Manages order lifecycle — create, track, cancel
  contact:
    name: Platform Team
    email: platform@company.com

servers:
  - url: https://api.company.com/orders/v1
    description: Production
  - url: http://localhost:8081
    description: Local development

paths:
  /orders:
    post:
      summary: Place a new order
      operationId: createOrder
      tags: [Orders]
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CreateOrderRequest'
      responses:
        "201":
          description: Order created successfully
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/OrderResponse'
        "400":
          $ref: '#/components/responses/ValidationError'
        "422":
          $ref: '#/components/responses/BusinessRuleViolation'

Code Generation 06

Generate Code from Specs

SpecKit can generate Spring Boot controller interfaces, DTOs, test stubs, and mock servers directly from your spec files.

bash
# Generate Spring Boot controller interfaces from spec
speckit generate --service order-service --target spring-boot --output src/main/java

# Generate DTOs (request/response classes)
speckit generate --service order-service --type dto --package com.company.order.dto

# Generate WireMock stubs for testing
speckit generate --service order-service --type mock --output src/test/resources/wiremock

# Generate Postman collection from spec
speckit generate --service order-service --type postman --output ./postman

# Generate all targets at once
speckit generate --all --target spring-boot

Validation 07

Validate & Lint Specs

SpecKit validates your specs against best practices and checks that your implementation matches what the spec says.

bash
# Lint all specs (style + completeness checks)
speckit lint --all

# Validate implementation matches spec (contract testing)
speckit contract-test --service order-service --env local

# Check for breaking changes against previous version
speckit diff --service order-service --baseline main

# Check cross-service schema compatibility
speckit compat --consumer payment-service --provider order-service
🤖 Prompt #7 — Review & Fix Spec Quality
Please review this OpenAPI spec and identify all quality issues: [PASTE YOUR openapi.yaml HERE] Check for: 1. Missing or incomplete descriptions on paths, parameters, and schemas 2. Missing error responses (400, 401, 403, 404, 422, 500) on each endpoint 3. Inconsistent naming conventions (camelCase vs snake_case in schemas) 4. Missing required field markers on request schemas 5. Endpoints that should have pagination but don't 6. Security definitions missing on endpoints that need auth 7. Missing examples in request/response schemas 8. Duplicate schema definitions that should use $ref For each issue found: - Quote the exact problematic section - Explain why it's a problem - Provide the corrected YAML Output the full corrected spec at the end.
