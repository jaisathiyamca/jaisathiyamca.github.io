Context Creation05

Build the Unified Context

Merge all 5 service context fragments into one queryable knowledge base that understands how the whole system fits together.

1

Merge all context fragments

SpecKit combines the frontend and all backend contexts into a single indexed file.

merge context
$ speckit context merge --all --output .speckit/context/unified.json ✓ Merged 5 context fragments → unified.json (4,088 tokens) ✓ Cross-service links resolved: 23 API call matches found ✓ Dependency graph built: 4 service-to-service relationships
2

Handle large codebases — chunking strategy

If your codebase is large (100k+ LOC), send it in focused chunks rather than all at once. SpecKit's chunking keeps context within AI token limits.

chunked scan for large codebase
$ speckit scan ./order-service --chunk-size 50 --chunk-by layer Chunk 1/4: Controllers layer → 42 files Chunk 2/4: Service layer → 38 files Chunk 3/4: Entity layer → 28 files Chunk 4/4: DTO layer → 51 files ✓ 4 chunks processed, context assembled
3

Export for AI prompt use

Export a markdown summary that you can paste directly into any AI prompt for context-aware queries.

export context
$ speckit context export --format markdown --role all ✓ .speckit/exports/context-full.md (ABA + Arch view) $ speckit context export --format markdown --role qa ✓ .speckit/exports/context-qa.md (endpoints + test data) $ speckit context export --format markdown --role dev ✓ .speckit/exports/context-dev.md (code structure view)

Context Creation06

Verify & Query Your Context

Before your team uses the context, run these verification checks to make sure it's accurate and complete.

5 Verification Queries to Run

bash — Verification Queries
# 1. List all discovered endpoints
speckit ask "List all API endpoints across all services with their HTTP methods"

# 2. Verify frontend-backend connections
speckit ask "Which frontend pages call which backend services?"

# 3. Find data ownership
speckit ask "Which service owns the Order entity? Where is it stored?"

# 4. Check service dependencies
speckit ask "Draw the service dependency graph — who calls who?"

# 5. Discover business rules
speckit ask "What business rules exist around order status transitions?"
✅
Context is good when: You can ask "what does the checkout flow do?" and get a coherent answer that matches what you know. If it's wrong or vague, re-run the scan on the service that owns that area.

They should get a 5% discount for scheduling more than 48 hours ahead. The business team wants a report showing all scheduled orders by delivery date." Please produce: 1. USER STORY As a [persona], I want [goal] so that [benefit] 2. ACCEPTANCE CRITERIA (Given/When/Then format, minimum 5) 3. IMPACT ANALYSIS — for each service in the system: - [FRONTEND_SERVICE]: What UI changes are needed? - [BE1_NAME]: What backend changes are needed? - [BE2_NAME]: What backend changes are needed? - [BE3_NAME]: What backend changes are needed? - [BE4_NAME]: What backend changes are needed? 4. NEW DATA FIELDS REQUIRED (name fields using the same naming convention found in the context) 5. BUSINESS RULES (numbered list) 6. OUT OF SCOPE (explicitly list what this story does NOT cover) 7. OPEN QUESTIONS (decisions needed before development starts) 8. TEAM ROUTING Which service team(s) need to be in the refinement session?
🤖 ABA Prompt #2 — Check if Feature Already Exists ABA

Is this capability (or part of it) ALREADY implemented in any service? Show evidence from the context. 2. Is there existing code/data that can be REUSED rather than rebuilt? 3. Are there any CONFLICTING existing rules that would prevent this feature? 4. What is the SMALLEST possible implementation (MVP) based on what already exists? 5.
