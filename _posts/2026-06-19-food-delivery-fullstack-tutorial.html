5Interview Question
?
What is the role of the Spring Boot Parent POM, and what problem does it solve compared to specifying every dependency version manually?

The spring-boot-starter-parent inherits from spring-boot-dependencies, which is a Bill of Materials (BOM) — a curated, pre-tested list of compatible library versions for everything in the Spring ecosystem. Without it, you'd have to manually pick compatible versions of Spring MVC, Hibernate, Jackson, Tomcat, SLF4J, and 50+ other transitive dependencies — and one wrong combination causes NoSuchMethodError or ClassCastException at runtime. With the parent POM, you declare a dependency without a version tag and Boot picks the tested-compatible version automatically. As an example: upgrading from Spring Boot 3.1 to 3.2 automatically upgrades Hibernate, Jackson, and Tomcat to their matching tested versions in one POM change.

1The Concept (TL;DR)

Spring Boot MVC splits the backend into three core jobs: Model (the data, e.g. a MenuItem), View (in a REST API, this is just the JSON response — React renders the real UI), and Controller (the traffic cop for HTTP requests). Production apps extend this into four layers: Controller → Service → Repository → Database, where each layer only ever talks to its direct neighbor.

  • Controller — handles HTTP, knows nothing about SQL.
  • Service — holds business rules, knows nothing about HTTP.
  • Repository — talks to the DB, knows nothing about business rules.
2Visual Explanation

A customer taps "Spice Garden" and the request travels down through the layers, then the data climbs back up:

FRAME 1 — Customer taps a restaurant in React 📱 axios.get("/api/restaurants/101/menu") ───────▶ FRAME 2 — Request lands at the Controller MenuController "I got a GET for restaurant 101. I don't fetch data myself — ask the Service." └──▶ menuService.getMenuForRestaurant(101) FRAME 3 — Service applies business rules MenuService "Is the restaurant open? Hide sold-out items." └──▶ menuRepository.findByRestaurantId(101) FRAME 4 — Repository queries the DB MenuRepository SELECT * FROM menu_items WHERE restaurant_id = 101; └──▶ 🗄 [ menu_items table ] FRAME 5 — Data climbs back up DB rows ──▶ Repository ──▶ Service (filters) ──▶ Controller (wraps in HTTP 200) FRAME 6 — Customer sees the menu 📱 🍛 Paneer Tikka ₹220 🍞 Butter Naan ₹40 Golden rule: data flows DOWN through layers on the way in, and back UP on the way out. No layer skips its neighbor.

MenuService.javaService Layer
@Service
public class MenuService {

    private final MenuRepository menuRepository;

    @Autowired
    public MenuService(MenuRepository menuRepository) {
        this.menuRepository = menuRepository;
    }

    public List<MenuItem> getMenuForRestaurant(Long restaurantId) {
        List<MenuItem> items = menuRepository.findByRestaurantId(restaurantId);

        // BUSINESS RULE lives here — never in Controller or Repository
        return items.stream()
                .filter(MenuItem::getAvailable)
                .collect(Collectors.toList());
    }
}

    public ResponseEntity<List<MenuItem>> getMenu(@PathVariable Long restaurantId) {

    }

1The Concept (TL;DR)

Dependency Injection (DI) means a class doesn't create its own dependencies — Spring hands them in. Spring keeps every managed object (a "Bean") in a giant registry called the ApplicationContext (the IoC — Inversion of Control — Container). When MenuController needs a MenuService, it doesn't write new MenuService() — it just declares "I need one" via the constructor, and Spring supplies an already-built instance.

Why this matters: it decouples classes from their dependencies' construction logic, which makes unit testing trivial (swap in a mock) and avoids the chaos of manually wiring hundreds of objects by hand.

2Visual Explanation

Think of the ApplicationContext as a head chef who has already hired and prepped every station before doors open:

FRAME 1 — App Boots Up Spring Boot starts ──▶ scans for @Component, @Service, @Repository, @RestController... FRAME 2 — Beans get created and stored in the Container ┌─────────────────────────────────────────────┐ │ ApplicationContext (IoC Container) │ │ │ │ [ MenuRepository bean ] │ │ [ MenuService bean ] ──depends on──▶ Repository │ │ [ MenuController bean ] ──depends on──▶ Service│ └─────────────────────────────────────────────┘ FRAME 3 — A request needs MenuController Spring: "I already built one. Here it is — fully wired, with its MenuService (which already has its MenuRepository) injected." FRAME 4 — Zero "new" keywords written by YOU You never wrote: new MenuRepository(); new MenuService(repo); new MenuController(service); Spring did all of that at boot time.
3Real-Time Example (The Food App)

MenuController needs MenuService, which needs MenuRepository, which needs a DataSource (DB connection pool). Instead of you wiring four objects by hand for every controller in the app — Orders, Payments, Delivery Tracking, Reviews — Spring builds the whole dependency graph once at startup and re-uses the same singleton instances everywhere they're needed.

1The Concept (TL;DR)

Data binding is how Spring automatically converts incoming JSON into a Java object (and vice versa for responses) using a library called Jackson. A DTO (Data Transfer Object) is a plain object shaped exactly like what the API needs to send or receive — distinct from your database Entity. You never expose Entities directly: it leaks DB schema, ties your API shape to your table structure, and risks exposing fields (like internal flags or other users' data) you never meant to send.

2Visual Explanation
FRAME 1 — React sends raw JSON { "restaurantId": 101, "items": [{"menuItemId":1,"qty":2}] } │ ▼ (Jackson deserializes) FRAME 2 — Spring binds JSON ──▶ DTO OrderRequestDTO ┌─────────────────────────┐ │ restaurantId: 101 │ │ items: [ {1, qty:2} ] │ └─────────────────────────┘ │ ▼ (Service maps DTO ──▶ Entity) FRAME 3 — Entity gets built for persistence Order (Entity) ┌─────────────────────────┐ │ id: null (auto-gen) │ │ status: PLACED │ │ totalPrice: 440.0 │ ◀── calculated, NOT trusted from client! │ createdAt: now() │ └─────────────────────────┘ │ ▼ saved to DB FRAME 4 — Response goes back out as A DIFFERENT DTO OrderResponseDTO { orderId: 555, status: "PLACED", eta: "32 min" } Notice: the client NEVER sees internal fields like restaurant payout %, or the raw Entity's JPA metadata.

1The Concept (TL;DR)

Without centralized exception handling, an error deep in your Service layer (like "restaurant doesn't exist") either crashes with an ugly 500-error stack trace leaking internals to the client, or forces you to write repetitive try/catch blocks in every single Controller method. TICKET #109

1The Concept (TL;DR)

JPA models relationships between Entities using annotations like @OneToMany and @ManyToOne, mirroring foreign keys. By default, collection relationships are lazy-loaded — Hibernate doesn't fetch them until you actually access the field — which avoids over-fetching, but creates a notorious trap: the N+1 query problem, where fetching N parent rows then lazily triggers N additional queries for their children, one at a time.

2Visual Explanation
THE N+1 TRAP — fetching 50 orders, each with its items List<Order> orders = orderRepository.findAll(); │ ▼ query #1 SELECT * FROM orders; ──▶ 50 rows returned for (Order o : orders) { o.getItems().size(); // accessing lazy collection } │ ▼ query #2, #3, #4 ... #51 (ONE per order!) (48 more nearly-identical queries) TOTAL: 51 round-trips to the database for ONE screen THE FIX — JOIN FETCH, one query total SELECT o, oi FROM Order o JOIN FETCH o.items oi │ ▼ query #1 (ONLY ONE) All 50 orders AND their items arrive together, already joined — Hibernate populates the in-memory object graph from a SINGLE result set.

Transactions & @Transactional — All-or-Nothing Order Placement

TICKET #110
Table: Order + Inventory Atomicity
@Transactional
Propagation
Isolation
1The Concept (TL;DR)

A transaction groups multiple database operations into one all-or-nothing unit: either every write succeeds and gets committed, or any failure rolls back every write in that unit, leaving the database exactly as it was. Spring's @Transactional annotation wraps a method in this guarantee automatically. Two settings matter most: propagation (does a method join an existing transaction or start a new one?) and isolation level (how much can concurrent transactions see of each other's uncommitted changes?).

2Visual Explanation
PLACING AN ORDER — 3 writes that must succeed TOGETHER @Transactional placeOrder(request) { 1. INSERT INTO orders (...) 2. INSERT INTO order_items (...) 3. UPDATE menu_items SET stock = stock - 1 } SCENARIO A — everything succeeds Step 1 ✓ Step 2 ✓ Step 3 ✓ ──▶ COMMIT — all 3 changes persist SCENARIO B — step 3 fails (item went out of stock mid-request) Step 1 ✓ Step 2 ✓ Step 3 ✗ ──▶ ROLLBACK Steps 1 and 2 are UNDONE too — the database NEVER ends up with an order record that has no matching stock deduction. Without @Transactional, you'd have a half-written order haunting your database forever. PROPAGATION — nested calls joining the SAME transaction OrderService.placeOrder() [starts transaction T1] └──▶ InventoryService.deduct() [REQUIRED: joins T1, doesn't start a new one] └──▶ AuditService.log() [REQUIRES_NEW: its own separate T2, commits even if T1 later rolls back — useful so audit logs survive failed orders]
3Real-Time Example (The Food App)

Placing an order writes to three tables. If the stock-deduction step fails because two customers raced for the last item, the entire order placement must roll back — we cannot end up with an "order placed" record pointing at an item that was never actually reserved. Conversely, an audit log entry recording "order placement attempted" might deliberately use its own separate transaction so it survives even when the order itself fails.

4Code Implementation

@Service
public class OrderService {

    // If ANY exception escapes this method, Spring rolls back
    // every database write made inside it — automatically.
    @Transactional
    public Order placeOrder(OrderRequestDTO request) {

        Order order = new Order();
        order.setStatus("PLACED");
        Order saved = orderRepository.save(order); // write #1

        for (OrderItemDTO itemDto : request.getItems()) {
            inventoryService.deductStock(itemDto.getMenuItemId(), itemDto.getQuantity());
            // write #2, #3... — joins the SAME transaction as placeOrder
            // (propagation REQUIRED, the default)
        }

        return saved;
        // If deductStock() throws (e.g. out of stock), Spring catches
        // the unchecked exception HERE and rolls back write #1 too.
    }
}

    // a brand new, independent one — this commits on its own,
    // regardless of what happens in the CALLER's transaction.
    @Transactional(propagation = Propagation.REQUIRES_NEW)

Isolation level examplePreventing a race on stock count
@Transactional(isolation = Isolation.REPEATABLE_READ)
public void deductStock(Long menuItemId, int quantity) {
    MenuItem item = menuItemRepository.findById(menuItemId).orElseThrow();

    if (item.getStock() < quantity) {
        throw new MenuItemUnavailableException(menuItemId);
    }
    item.setStock(item.getStock() - quantity);
    // REPEATABLE_READ means: once THIS transaction reads stock,
    // no other transaction's concurrent write can change what we
    // see for the rest of this transaction — guards against two
    // customers both seeing "1 item left" and both purchasing it.
}

WITHOUT AOP — logging duplicated in every Service method placeOrder() { log.info("start"); ...logic...; log.info("end"); } cancelOrder() { log.info("start"); ...logic...; log.info("end"); } updateStatus() { log.info("start"); ...logic...; log.info("end"); } (same boilerplate, copy-pasted N times across the codebase) WITH AOP — written ONCE, applied everywhere automatically ┌─────────────────────────────────────────────┐ │ @Aspect LoggingAspect │ │ Pointcut: "any method inside com.fooddelivery.service.*"│ │ Advice: "log before AND after, with timing" │ └───────────────────┬───────────────────────────┘ │ Spring creates a PROXY around every │ matching bean automatically ▼ Client calls orderService.placeOrder(request) │ ▼ call is intercepted by the PROXY first [Aspect: log "Entering placeOrder"] ▼ REAL placeOrder() logic runs ▼ [Aspect: log "Exiting placeOrder, took 42ms"] ▼ result returned to client OrderService.java itself has ZERO logging code in it. The aspect applies to every Service automatically.
3Real-Time Example (The Food App)

We want to log entry/exit and execution time for every Service method, enforce that only authenticated restaurant-owner accounts can call certain RestaurantService methods, and retry transient payment-gateway failures — all without scattering if checks and try/catch blocks across every business method. LoggingAspect.java@Around advice with timing
@Aspect
@Component
public class LoggingAspect {

    private static final Logger log = LoggerFactory.getLogger(LoggingAspect.class);

    // Pointcut expression: matches every public method inside
    // any class in the service package
    @Around("execution(* com.fooddelivery.service.*.*(..))")
    public Object logExecutionTime(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        String methodName = joinPoint.getSignature().toShortString();

        log.info("Entering {}", methodName);
        Object result = joinPoint.proceed(); // runs the ACTUAL method
        long duration = System.currentTimeMillis() - start;
        log.info("Exiting {} — took {}ms", methodName, duration);

        return result;
    }
}

What's the practical difference between @Before, @After, and @Around advice in Spring AOP, and when would @Around be necessary instead of the simpler two?

Expect: @Before and @After run code at fixed points without controlling the original method's execution or return value. @Around is strictly more powerful — it receives a ProceedingJoinPoint, can decide whether to call proceed() at all (enabling caching-style "skip the real method" logic), can modify the arguments before proceeding, and can modify or replace the return value — which is exactly why timing, caching, and retry logic all require @Around.

TICKET #112

1The Concept (TL;DR)

A JWT (JSON Web Token) is a self-contained, signed token that proves who a user is without the server needing to store session state. After login, the server signs a token containing the user's ID/role and a secret key; the client sends that token on every subsequent request; the server verifies the signature (not a database lookup) to confirm it's authentic and unexpired. Spring Security intercepts every request through a filter chain before it ever reaches your Controller.

4Code Implementation
JwtService.javaIssuing and validating tokens
@Service
public class JwtService {

    @Value("${jwt.secret}") // NEVER hardcode this — comes from env config
    private String secretKey;

    public String generateToken(User user) {
        return Jwts.builder()
            .setSubject(user.getId().toString())
            .claim("role", user.getRole())
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + 3600_000)) // 1hr
            .signWith(SignatureAlgorithm.HS256, secretKey)
            .compact();
    }

    public Claims validateAndExtractClaims(String token) {
        try {
            return Jwts.parser().setSigningKey(secretKey)
                .parseClaimsJws(token).getBody();
        } catch (JwtException e) {
            throw new InvalidTokenException("Token invalid or expired");
        }
    }
}
JwtAuthFilter.javaRuns once per request, before Controllers
@Component
public class JwtAuthFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain chain)
            throws ServletException, IOException {

        String header = request.getHeader("Authorization");
        if (header != null && header.startsWith("Bearer ")) {
            String token = header.substring(7);
            try {
                Claims claims = jwtService.validateAndExtractClaims(token);
                // Populate Spring Security's context — Controllers and
                // AOP-based @PreAuthorize checks can now see "who is this?"
                UsernamePasswordAuthenticationToken auth =
                    new UsernamePasswordAuthenticationToken(
                        claims.getSubject(), null,
                        List.of(new SimpleGrantedAuthority(claims.get("role", String.class))));
                SecurityContextHolder.getContext().setAuthentication(auth);
            } catch (InvalidTokenException e) {
                // leave context empty — Spring Security will reject the
                // request downstream if the endpoint requires auth
            }
        }
        chain.doFilter(request, response); // pass to the next filter / Controller
    }
}

JWTs are commonly described as "stateless," but how would you handle logging a user out or revoking a compromised token before its natural expiry, given that the server never stored the token in the first place?

This is a genuinely hard trade-off interviewers probe for: true statelessness means the server CAN'T simply "delete" a token it never stored. Resilience4j — Surviving a Flaky Payment Gateway

TICKET #114
Table: Backup Generator Room
Retry
Circuit Breaker
Fallback
1The Concept (TL;DR)

External dependencies fail — a third-party payment gateway times out occasionally under normal load, or goes fully down during an outage. Retry automatically re-attempts a failed call (good for brief, transient blips). A Circuit Breaker detects sustained failure and "trips open," stopping further calls for a cooldown period entirely — preventing a struggling downstream service from being hammered with retries while it's already failing, and preventing your own service from piling up slow, doomed requests.

2Visual Explanation
CIRCUIT BREAKER STATE MACHINE CLOSED (normal operation — calls flow through) │ │ failure rate exceeds threshold (e.g. 50% of last 10 calls failed) ▼ OPEN (circuit "trips" — calls REJECTED INSTANTLY, no network call even attempted, for a cooldown window) │ │ after cooldown (e.g. 30 seconds) ▼ HALF_OPEN (cautiously allow a FEW test calls through) │ ├── test calls succeed ──▶ back to CLOSED └── test calls still fail ──▶ back to OPEN, wait again While OPEN, your service fails FAST (no waiting on a timeout) and can serve a fallback response instead — protecting both itself and the struggling downstream service from being overwhelmed by retries.
3Real-Time Example (The Food App)

The Payment Service occasionally times out for a few seconds under load — Retry handles that gracefully. But if it goes fully down for 10 minutes during an incident, we don't want every single checkout attempt across the platform to hang waiting for a timeout that's guaranteed to fail; the Circuit Breaker trips open, checkout fails fast with a clear "payments temporarily unavailable" message, and the system automatically starts probing again once the gateway recovers.

PaymentService.javaAnnotated with Retry + CircuitBreaker + fallback
@Service
public class PaymentService {

    // Order matters: Retry runs INSIDE the circuit breaker's view —
    // failed retries still count toward tripping the breaker open.
    @CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
    @Retry(name = "paymentService")
    public PaymentResultDTO chargeCard(ChargeRequestDTO request) {
        return paymentGatewayClient.charge(request); // the flaky external call
    }

    // MUST match the original method's signature, plus a Throwable param
    public PaymentResultDTO paymentFallback(ChargeRequestDTO request, Throwable t) {
        log.warn("Payment gateway unavailable, circuit open: {}", t.getMessage());
        return PaymentResultDTO.failedWithReason("PAYMENT_TEMPORARILY_UNAVAILABLE");
        // Checkout fails FAST with a clear message — instead of every
        // request hanging on a doomed network call for 10+ seconds each
    }
}
5Interview Question
?
Why is blindly retrying a failed payment charge dangerous, and how would you make a retried payment operation safe?

This tests whether the candidate connects resilience patterns to idempotency: if the first charge attempt actually succeeded on the gateway's side but the response was lost in transit (e.g. a network blip after the charge went through), a naive retry could double-charge the customer. The fix is an idempotency key — a unique ID generated client-side and sent with the request — so the payment gateway recognizes a retried request as "already processed" and returns the original result instead of charging again.

5Interview Question

Why is Vite faster than Create React App (Webpack), and what is the difference between a dev server and a production build?

Create React App uses Webpack, which bundles the entire application on startup — even if you only changed one file, it re-processes and re-bundles everything. On large projects this takes 30–60 seconds on every start. Vite does two things differently: (1) it serves files as native ES Modules directly to the browser in development — no bundling at all, just transform-on-demand — so startup is near-instant regardless of project size. (2) Hot Module Replacement (HMR) updates only the changed module, not the whole bundle. 1The Concept (TL;DR)

When state changes, React doesn't tear down and rebuild the actual browser DOM (which is slow). 1The Concept (TL;DR)

Every re-render re-runs the entire component function from scratch — including any expensive calculations and any function definitions inside it. useMemo caches the result of a calculation, only recomputing it when its dependencies change. useCallback caches a function reference itself, which matters because a new function is created on every render by default — and that breaks reference-equality checks used by React.memo (next lesson) on child components.

import { useState, useCallback } from "react";

function RestaurantPage({ menuItems }) {

  // WITHOUT useCallback: a NEW function object is created on
  // every render, breaking React.memo on MenuItemCard below.

  // across renders, as long as its dependencies don't change.
  const handleAddToCart = useCallback((item) => {

      if (existing) {
        return prev.map((i) => i.id === item.id ? { ...i, qty: i.qty + 1 } : i);
      }
      return [...prev, { ...item, qty: 1 }];
    });
  }, []); // empty deps — this function never needs to change identity

  return <MenuList items={menuItems} onAddToCart={handleAddToCart} />;
}

  step: "ADDRESS",
  address: null,
  paymentMethod: null,
  isSubmitting: false,
  error: null,
};

// A pure function: same (state, action) input ALWAYS produces
// the same output — no side effects, no API calls inside here.
function checkoutReducer(state, action) {
  switch (action.type) {
    case "SUBMIT_ADDRESS":
      return { ...state, address: action.payload, step: "PAYMENT", error: null };

    case "START_PAYMENT":
      return { ...state, isSubmitting: true, error: null };

    case "PAYMENT_SUCCESS":
      return { ...state, isSubmitting: false, step: "CONFIRMATION" };

    case "PAYMENT_FAILED":
      return { ...state, isSubmitting: false, error: action.payload };

    default:
      throw new Error(`Unknown action: ${action.type}`);
  }
}

hooks/useFetch.jsA reusable data-fetching Hook
import { useState, useEffect } from "react";
import apiClient from "../api/axiosClient";

// Custom Hook: a plain function, named with "use", that
// internally calls OTHER hooks (useState, useEffect)
export function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let cancelled = false; // guard against setting state after unmount

    async function load() {
      setLoading(true);
      try {
        const response = await apiClient.get(url);
        if (!cancelled) setData(response.data);
      } catch (err) {
        if (!cancelled) setError(err);
      } finally {
        if (!cancelled) setLoading(false);
      }
    }
    load();

    return () => { cancelled = true; }; // cleanup if url changes or unmounts
  }, [url]);

  return { data, loading, error };
}

What are the "Rules of Hooks," and why does breaking them (e.g. calling a Hook inside an if-statement) cause real bugs rather than just being a style violation?

The rules: only call Hooks at the top level (never inside loops, conditions, or nested functions), and only call them from React function components or other custom Hooks. ErrorBoundary.jsxMust be a class component — no Hook equivalent exists
import { Component } from "react";

// Error Boundaries are one of the few remaining cases that
// REQUIRE a class component — there is no Hook for this yet.
class ErrorBoundary extends Component {
  constructor(props) {
    super(props);
    this.state = { hasError: false };
  }

  // React calls this automatically when a CHILD throws during render
  static getDerivedStateFromError(error) {
    return { hasError: true };
  }

  componentDidCatch(error, errorInfo) {
    // log to an error-tracking service (e.g. Sentry) here
    console.error("Caught by ErrorBoundary:", error, errorInfo);
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback;
    }
    return this.props.children; // normal rendering when nothing has gone wrong
  }
}

export default ErrorBoundary;

PricingConfig.javaClassic / textbook Singleton
public class PricingConfig {

    // volatile + double-checked locking: thread-safe lazy init
    private static volatile PricingConfig instance;

    private final double taxRate;
    private final double baseDeliveryFee;

    // PRIVATE constructor — nobody outside this class can "new" it
    private PricingConfig() {
        this.taxRate = 0.05;
        this.baseDeliveryFee = 30.0;
    }

    public static PricingConfig getInstance() {
        if (instance == null) {
            synchronized (PricingConfig.class) {
                if (instance == null) { // double-checked
                    instance = new PricingConfig();
                }
            }
        }
        return instance;
    }

    public double getTaxRate() { return taxRate; }
    public double getBaseDeliveryFee() { return baseDeliveryFee; }
}

How does Decorator differ from Inheritance for extending behaviour, and when does it win?

1The Concept (TL;DR)

When multiple threads access shared mutable state simultaneously without coordination, you get race conditions — intermittent bugs that destroy data integrity. InventoryService.javaThread-safe stock reservation
@Service
public class InventoryService {
    // ReentrantLock gives us tryLock() which synchronized can't
    private final ReentrantLock stockLock = new ReentrantLock();
    private final Map<Long, Integer> menuItemStock = new ConcurrentHashMap<>();

    public boolean reserveItems(List<OrderItem> items) {
        stockLock.lock(); // only one thread reserves at a time
        try {
            // Phase 1: check all items have enough stock
            for (OrderItem item : items) {
                int available = menuItemStock.getOrDefault(item.getMenuItemId(), 0);
                if (available < item.getQuantity()) return false;
            }
            // Phase 2: decrement all (only reached if all checks passed)
            for (OrderItem item : items) {
                menuItemStock.merge(item.getMenuItemId(), -item.getQuantity(), Integer::sum);
            }
            return true;
        } finally {
            stockLock.unlock(); // ALWAYS release — even if exception thrown
        }
    }
}

What is a deadlock and how would you prevent one in the inventory service above?

A deadlock occurs when Thread A holds Lock1 and waits for Lock2, while Thread B holds Lock2 and waits for Lock1 — both block forever. In inventory, if two services each lock separate resource maps in different orders, they can deadlock. Prevention strategies: (1) lock ordering — always acquire locks in the same predefined order (e.g., lock by menuItemId ascending), (2) tryLock with timeout — lock.tryLock(500, MILLISECONDS) returns false instead of blocking indefinitely, (3) eliminate shared mutable state — use database row-level locks (SELECT FOR UPDATE) instead of in-memory locks so the DB handles contention, which is what production systems do. In Spring, @Lock(LockModeType.PESSIMISTIC_WRITE) on a repository method achieves this.

Database Indexing & Query Optimization — Making Queries 1000× Faster

TICKET #310

Index Design Rules
1The Concept (TL;DR)

A database index is a separate data structure (B-Tree by default in PostgreSQL/MySQL) that keeps a sorted copy of one or more columns, with pointers back to the full rows. Without an index, a query scans every row — O(n) full table scan. With an index, the DB performs a B-Tree traversal — O(log n). On a table with 10 million orders, a full scan might take 8 seconds; an index lookup takes 2 milliseconds. The trade-off: indexes consume disk space and slow down INSERT/UPDATE (the index must also be updated). Never index everything — index what you query.

1The Concept (TL;DR)

Normalization is the process of structuring a relational database to eliminate data redundancy and prevent update anomalies. Each normal form eliminates a specific type of dependency problem. ACID (Atomicity, Consistency, Isolation, Durability) is the set of guarantees a relational database makes about transactions — what separates a real database from a file. SELECT stock FROM menu_items WHERE id = 5; -- still returns 1 (snapshot)

When would you intentionally denormalize a table in a production system?

Normalization optimizes for write correctness; denormalization trades write complexity for read speed. RateLimiter.javaToken Bucket — single node, in-memory
public interface RateLimiter {
    boolean allowRequest(String clientId);
}

public class TokenBucketRateLimiter implements RateLimiter {
    private final int capacity;         // max tokens in bucket
    private final double refillRate;     // tokens per second
    private final Map<String, Bucket> buckets = new ConcurrentHashMap<>();

    static class Bucket {
        double tokens;
        long lastRefillNanos;
        Bucket(double capacity) { tokens = capacity; lastRefillNanos = System.nanoTime(); }
    }

    public TokenBucketRateLimiter(int capacity, double refillRate) {
        this.capacity = capacity;
        this.refillRate = refillRate;
    }

    @Override
    public synchronized boolean allowRequest(String clientId) {
        Bucket bucket = buckets.computeIfAbsent(clientId, k -> new Bucket(capacity));
        refill(bucket);
        if (bucket.tokens >= 1.0) {
            bucket.tokens -= 1.0;
            return true;
        }
        return false; // 429 Too Many Requests
    }

    private void refill(Bucket bucket) {
        long now = System.nanoTime();
        double elapsed = (now - bucket.lastRefillNanos) / 1_000_000_000.0;
        bucket.tokens = Math.min(capacity, bucket.tokens + elapsed * refillRate);
        bucket.lastRefillNanos = now;
    }
}

For Token Bucket: store tokens and lastRefill as a Redis Hash per client key; use a Lua script to make the "refill + check + decrement" operation atomic (Redis is single-threaded, Lua scripts run atomically). Alternatively, use the sliding window counter with Redis INCR + EXPIRE: INCR rate:clientId:window and EXPIRE rate:clientId:window 60. The key trade-off: Redis adds ~1ms network round-trip per request — for truly latency-sensitive paths, you can approximate with per-instance counters + a sync heartbeat, accepting slight over-counting as a known trade-off.

    private final PaymentService payment;
    private final OrderService orderSvc;
    private final NotificationService notification;
    private final PromoService promo;

public interface OrderCommand {
    OrderCommandResult execute();
    void undo();
}

public class CancelOrderCommand implements OrderCommand {
    private final Order order;
    private String refundRef; // stored for undo

    @Override
    public OrderCommandResult execute() {
        inventory.releaseItems(order.getItems());
        refundRef = payment.refund(order.getPaymentRef(), order.getTotal());
        order.setStatus("CANCELLED");
        orderRepo.save(order);
        notification.sendCancellation(order);
        return OrderCommandResult.success("Order cancelled, refund: " + refundRef);
    }

    @Override
    public void undo() {
        // reverse the cancellation (e.g., restaurant calls back within 30 sec)
        payment.reverseRefund(refundRef);
        inventory.reserveItems(order.getItems());
        order.setStatus("CONFIRMED");
        orderRepo.save(order);
    }
}

// Invoker: maintains history, enables undo
@Service
public class OrderCommandInvoker {
    private final Deque<OrderCommand> history = new ArrayDeque<>();

    public OrderCommandResult execute(OrderCommand cmd) {
        OrderCommandResult result = cmd.execute();
        history.push(cmd);
        return result;
    }

    public void undoLast() {
        if (!history.isEmpty()) history.pop().undo();
    }
}

Redis Caching — Speeding Up the Restaurant List

TICKET #403
Table: Specials Board (Quick Glance)

1The Concept (TL;DR)

Redis is an in-memory key-value store, dramatically faster than querying a relational database because it skips disk I/O entirely. The cache-aside pattern: on a read, check Redis first; if the data's there (a cache hit), return it immediately; if not (a cache miss), query the database, then store the result in Redis for next time, with a TTL (time-to-live) so stale data eventually expires.

2Visual Explanation
FRAME 1 — First customer in Koramangala opens the app GET /api/restaurants?area=Koramangala │ ▼ Redis.get("restaurants:Koramangala") ──▶ MISS (not cached yet) │ ▼ fall through to the database SELECT * FROM restaurants WHERE area = 'Koramangala' │ ▼ (slow-ish: ~80ms) Redis.set("restaurants:Koramangala", results, ttl=300s) │ ▼ Response sent: 35 restaurants, took ~85ms total FRAME 2 — Second customer, 10 seconds later, same area GET /api/restaurants?area=Koramangala │ ▼ Redis.get("restaurants:Koramangala") ──▶ HIT! (~2ms) │ ▼ DATABASE NEVER TOUCHED Response sent: same 35 restaurants, took ~3ms total FRAME 3 — 5 minutes later, TTL expires Redis automatically evicts the key. Next request becomes a MISS again, refreshing the cache with current data. This is the trade-off: faster reads, in exchange for data that can be up to `ttl` seconds stale.
3Real-Time Example (The Food App)

The restaurant listing for a given area doesn't change every second — it's a perfect caching candidate. Order status, on the other hand, changes constantly and needs near-real-time accuracy, so it's a poor caching candidate (or needs a very short TTL plus active invalidation on status change).

4Code Implementation
RestaurantService.javaCache-aside pattern, Spring Data Redis
@Service
public class RestaurantService {

    private final RestaurantRepository repository;
    private final RedisTemplate<String, List<Restaurant>> redisTemplate;

    public List<Restaurant> getRestaurantsByArea(String area) {
        String cacheKey = "restaurants:" + area;

        // 1. Check the cache first
        List<Restaurant> cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return cached; // CACHE HIT — database never touched
        }

        // 2. Cache miss — fall through to the database
        List<Restaurant> fromDb = repository.findByArea(area);

        // 3. Populate the cache for next time, with a 5-minute TTL
        redisTemplate.opsForValue().set(
            cacheKey, fromDb, Duration.ofMinutes(5));

        return fromDb;
    }
}

CAP Theorem & Consistency Models — Why You Can't Have Everything

TICKET #406
Table: The Three-Star Dilemma
CAP Triangle

1The Concept (TL;DR)

The CAP Theorem (Brewer, 2000) states that a distributed system can guarantee at most two of three properties simultaneously: Consistency (every read sees the most recent write or an error), Availability (every request gets a non-error response), Partition tolerance (the system continues operating when network partitions occur). Since network partitions are inevitable in distributed systems, the real choice is between CP (sacrifice availability during partition) and AP (sacrifice consistency during partition). Modern systems (DynamoDB, Cassandra) further nuance this with tunable consistency — you pick your trade-off per operation.

A customer's order shows "DELIVERED" in the app but the restaurant's dashboard still shows it as "OUT_FOR_DELIVERY". Is this a CAP violation? How would you fix it?

This is an AP trade-off made visible to the user — not a CAP "violation" (CAP is a theorem about what's possible, not a bug). The system chose availability (always respond) + partition tolerance over strong consistency, resulting in two nodes seeing different state briefly. A single PostgreSQL instance hits its ceiling — one machine's disk, CPU, and connection pool can only scale so far. The database scaling playbook: Replication (copy data to standby nodes for fault tolerance), Read Replicas (direct read traffic to replicas, writes only to primary — horizontal read scaling), and Sharding (split data across multiple primary databases by a shard key — horizontal write scaling). Each solves a different bottleneck and introduces different trade-offs.

What is a "hot shard" and how do you fix it?

A hot shard occurs when the shard key distributes data unevenly — e.g., sharding by restaurant_id means Domino's (with 10× more orders than any other restaurant) lands on one shard, which gets overwhelmed while others are idle. Fixes: (1) Better shard key — choose high-cardinality, uniformly-distributed keys like customer_id or a UUID-based order ID. (2) Composite shard key — shard by customer_id + date to distribute hot customers across time buckets. (3) Consistent hashing (next lesson) — virtual nodes mean one "hot" entity is split across multiple physical shards. (4) Application-level mitigation — cache popular restaurant data aggressively in Redis so the DB shard isn't directly hit for every read. In practice, hot shards are detected via DB monitoring (CPU/IOPS per shard) and trigger re-sharding with consistent hashing.

Consistent Hashing — Adding Servers Without Reshuffling Everything

TICKET #408
Table: The Elastic Kitchen
The Problem
Hash Ring
Virtual Nodes
1The Concept (TL;DR)

With naive modulo sharding (key % N), adding or removing one server changes the assignment of almost every key, causing a massive data migration. Consistent hashing places both servers and keys on a hash ring (0 to 2³²-1 in a circle). Each key is assigned to the first server clockwise from its hash position. When a server is added/removed, only the keys between the new server and its predecessor need to move — O(K/N) keys instead of O(K). Used by: DynamoDB, Cassandra, Redis Cluster, Akamai CDN, Memcached clusters.

2Visual Explanation
2Visual Explanation
HASH RING (0 ... 2^32 circular): 0 / 2^32 │ S1 ──┤ (hash(Server1) = position on ring) │ S3 ───────┤───── Key: order_99 → assigned to S3 (next clockwise) │ S2 ──┤ (hash(Server2)) │ Add Server S4 between S1 and S3: ┌─────────────────────────────────────────────────┐ │ Before: order_99 → S3 │ │ After S4 added between S1 and order_99: │ │ order_99 → S4 (only this key moves) │ │ All other keys: unchanged ✓ │ └─────────────────────────────────────────────────┘ VIRTUAL NODES — solve hot spots: Without virtual nodes: S1 may "own" a large arc → more data With virtual nodes (e.g., 150 vNodes per server): S1 → vNode_S1_1, vNode_S1_2, ... vNode_S1_150 (spread across ring) S2 → vNode_S2_1, vNode_S2_2, ... vNode_S2_150 Result: each server owns many small arcs → uniform distribution Adding S4: ~1/4 of vNodes reassigned = ~1/4 of keys move → smooth rebalance DynamoDB and Cassandra both use virtual nodes (vnodes) internally.

The food delivery platform's Redis cache cluster starts with 3 nodes. During a festival surge (Diwali), 2 more Redis nodes are added. With naive hash(key) % 3 → hash(key) % 5, almost all cache keys map to different nodes — instant 100% cache miss rate, thundering herd on the database. With consistent hashing, only ~40% of keys move to the new nodes, maintaining ~60% cache hit rate during the transition.

ConsistentHashRouter.javaHash ring with virtual nodes
public class ConsistentHashRouter {
    private final TreeMap<Long, String> ring = new TreeMap<>();
    private static final int VIRTUAL_NODES = 150;

    public void addServer(String serverName) {
        for (int i = 0; i < VIRTUAL_NODES; i++) {
            long hash = hash(serverName + "-vnode-" + i);
            ring.put(hash, serverName);
        }
    }

    public void removeServer(String serverName) {
        for (int i = 0; i < VIRTUAL_NODES; i++) {
            ring.remove(hash(serverName + "-vnode-" + i));
        }
    }

    public String getServer(String key) {
        if (ring.isEmpty()) throw new IllegalStateException("No servers");
        long hash = hash(key);
        // find first server clockwise from key's position on ring
        Map.Entry<Long, String> entry = ring.ceilingEntry(hash);
        if (entry == null) entry = ring.firstEntry(); // wrap around
        return entry.getValue();
    }

    private long hash(String key) {
        // MurmurHash3 or MD5 for uniform distribution
        CRC32 crc = new CRC32();
        crc.update(key.getBytes(StandardCharsets.UTF_8));
        return crc.getValue();
    }
}

If a cache node fails and comes back up after 10 minutes, what happens to the data it was serving, and how do consistent hashing + replication handle this?

When node S3 fails: all keys previously on S3 are now routed to S4 (the next clockwise node). Requests to S4 for those keys are cache misses → DB lookups → cache population on S4. When S3 recovers: keys route back to S3 but S3's cache is cold → another wave of cache misses. Fix: replication — each key is written to N=3 nodes (coordinator + 2 replicas clockwise). When S3 was down, S4 and S5 were receiving all writes. When S3 comes back, it runs a read repair or anti-entropy process (Cassandra's Merkle tree comparison) to sync missed writes from its neighbors. This is exactly what Cassandra does — hinted handoff queues writes for unavailable nodes, replaying them on recovery.

What is a "cache stampede" (thundering herd) and how do you prevent it?

A cache stampede happens when a popular cached item expires and thousands of concurrent requests all get cache misses simultaneously, hammering the database with the same query at once. Prevention strategies: (1) Mutex lock on cache miss — only one request populates the cache, others wait and then read the cached value (using Redis SET NX as a distributed lock). (2) Probabilistic early expiration (XFetch algorithm) — recompute the cache slightly before it expires based on the time-to-compute vs remaining TTL, so expiration is never simultaneous. (3) Background refresh — a scheduled job proactively refreshes popular cache keys before they expire, so the cache is never truly empty. (4) Stale-while-revalidate — serve stale data immediately while asynchronously refreshing in the background (Spring Cache's @CachePut on a background thread). The correct answer for "top restaurants in city" on a food delivery app is background refresh — the data changes every few minutes at most, so keeping it warm is trivial.

Saga Pattern — Distributed Transactions Without 2PC

TICKET #410
Table: The Multi-Kitchen Coordination
Why not 2PC
Choreography
Orchestration
1The Concept (TL;DR)

In a microservices system, a single business operation (place order) spans multiple services, each with its own database. You can't use a traditional DB transaction across services. 2PC (Two-Phase Commit) exists but is slow, blocking, and a single coordinator failure can lock all participants. The Saga pattern replaces a distributed transaction with a sequence of local transactions, each publishing an event that triggers the next. On failure, compensating transactions roll back completed steps. Two styles: Choreography (each service reacts to events from others — no central coordinator) and Orchestration (a central Saga Orchestrator commands each service step by step).

2Visual Explanation
SAGA — Order Placement across 4 microservices: HAPPY PATH (Choreography): OrderService InventoryService PaymentService NotificationService │ │ │ │ createOrder() │ │ │ │──ORDER_CREATED──▶ │ │ │ │ reserveItems() │ │ │ │──ITEMS_RESERVED─▶ │ │ │ │ chargeCard() │ │ │ │──PAYMENT_SUCCESS─▶ │ │ │ │ sendSMS() FAILURE PATH — Payment fails → compensate backwards: │ │ │ │ │ chargeCard() FAILS │ │ │ │ │◀─PAYMENT_FAILED──│ │ releaseItems() ← compensating tx │◀──ITEMS_RELEASED───│ cancelOrder() ← compensating tx Status: ORDER_CANCELLED KEY INSIGHT: Each step is a local DB transaction. Compensation undoes committed steps. No global lock ever held across services.
3Real-Time Example (The Food App)

Placing an order in the food delivery platform: (1) OrderService creates order record (local tx), publishes ORDER_PLACED. (2) InventoryService listens, reserves items, publishes ITEMS_RESERVED. (3) PaymentService listens, charges card, publishes PAYMENT_SUCCESS. (4) KitchenService listens, dispatches order. If payment fails: PaymentService publishes PAYMENT_FAILED, InventoryService listens and releases items, OrderService listens and marks order cancelled. No 2PC, no blocking locks, full audit trail in the event log.

    // Step 1: initiate saga — called by OrderService REST handler
    @Transactional
    public void startSaga(Order order) {
        sagaStateRepo.save(new SagaState(order.getId(), "RESERVING_INVENTORY"));
        kafka.send("inventory-commands", new ReserveItemsCommand(order));
    }

    // Step 2: inventory replied — advance or compensate
    @KafkaListener(topics = "inventory-replies")
    @Transactional
    public void onInventoryReply(InventoryReply reply) {
        SagaState saga = sagaStateRepo.findByOrderId(reply.getOrderId());
        if (reply.isSuccess()) {
            saga.setState("CHARGING_PAYMENT");
            sagaStateRepo.save(saga);
            kafka.send("payment-commands", new ChargeCardCommand(reply));
        } else {
            saga.setState("FAILED_INSUFFICIENT_STOCK");
            sagaStateRepo.save(saga);
            kafka.send("order-commands", new CancelOrderCommand(saga.getOrderId()));
        }
    }

    // Step 3: payment replied
    @KafkaListener(topics = "payment-replies")
    @Transactional
    public void onPaymentReply(PaymentReply reply) {
        SagaState saga = sagaStateRepo.findByOrderId(reply.getOrderId());
        if (reply.isSuccess()) {
            saga.setState("COMPLETED");
            sagaStateRepo.save(saga);
            kafka.send("kitchen-commands", new DispatchKitchenCommand(reply));
        } else {
            // Compensate: release inventory
            saga.setState("COMPENSATING");
            sagaStateRepo.save(saga);
            kafka.send("inventory-commands", new ReleaseItemsCommand(saga));
        }
    }

5Interview Question

What is "dual-write" and why is it dangerous in a Saga? How do you fix it?

Dual-write: updating the DB and publishing to Kafka in two separate operations within the same method. The danger: if the app crashes after the DB commit but before the Kafka send, the event is lost — the Saga never advances, the order is stuck with no compensating path. Fix: Transactional Outbox Pattern — instead of publishing to Kafka directly, write the event as a row in an outbox table in the same local DB transaction as the business data. A separate process (Debezium CDC or a polling service) reads committed outbox rows and publishes to Kafka, then marks them sent. Guarantees: the event is only published if the DB transaction committed, and it's published at least once (idempotent consumers handle duplicates). This is the standard production solution for reliable event publishing in microservices.

5Interview Question

What is the difference between p50, p95, and p99 latency, and why does the average latency metric lie?

Average (mean) latency masks outliers: if 99% of requests complete in 10ms but 1% take 5000ms, the average might be 60ms — looks fine, but 1 in 100 customers has a terrible experience. Percentiles tell the truth: p50 (median) = 50% of requests are faster than this — the "typical" experience. p95 = 95% of requests are faster — what most users experience. p99 = 99% are faster — your worst typical case (not outliers, not anomalies — this is what your SLA should be measured against). A system with p50=10ms and p99=4000ms has a serious tail latency problem — often caused by GC pauses, connection pool exhaustion, or a slow external dependency. Always set SLOs on p99, alert on p99, and optimize for p99 in production.

public class PaymentService {

    public PaymentResult chargeWithIdempotency(Order order) {
        // Idempotency key: same key = Stripe returns same result, no double charge
        String idempotencyKey = "order-" + order.getId() + "-v1";

        // Check if we already processed this payment (after retry)
        Optional<PaymentRecord> existing = paymentRepo.findByIdempotencyKey(idempotencyKey);
        if (existing.isPresent()) {
            log.info("Idempotent: returning cached payment result for order {}", order.getId());
            return existing.get().toResult();
        }

        // Call Stripe with the same key — Stripe deduplicates on their side too
        StripeCharge charge = stripe.charges().create(
            ChargeCreateParams.builder()
                .setAmount((long)(order.getTotal() * 100))
                .setCurrency("inr")
                .setSource(order.getPaymentToken())
                .setIdempotencyKey(idempotencyKey)
                .build()
        );

        PaymentRecord record = new PaymentRecord(idempotencyKey, charge.getId(), order.getId());
        paymentRepo.save(record);
        return PaymentResult.success(charge.getId());

public class DriverMatchingService {
    private final RedisTemplate<String, String> redis;

    // Drivers update location every 10 seconds (GPS app)
    public void updateDriverLocation(Long driverId, double lat, double lon) {
        redis.opsForGeo().add("drivers:available",
            new Point(lon, lat), driverId.toString());
    }

    // Find nearest available driver within 5km of restaurant
    public Optional<Long> findNearestDriver(double restaurantLat, double restaurantLon) {
        GeoResults<RedisGeoCommands.GeoLocation<String>> results =
            redis.opsForGeo().radius("drivers:available",
                new Circle(new Point(restaurantLon, restaurantLat),
                            new Distance(5, Metrics.KILOMETERS)),
                GeoRadiusCommandArgs.newGeoRadiusArgs().sortAscending().limit(1));

        if (results == null || results.getContent().isEmpty()) return Optional.empty();
        String driverId = results.getContent().get(0).getContent().getName();
        return Optional.of(Long.parseLong(driverId));
    }

Walk me through how your system handles a complete outage of the payment gateway (Stripe is down for 10 minutes). What happens to in-flight orders, and what's your recovery strategy?

This is the failure-mode question that separates senior engineers. Answer: (1) Resilience4j Circuit Breaker on the Stripe call opens after 5 consecutive failures — new payment calls immediately return a fallback error rather than timing out. (2) In-flight orders that already reserved inventory but haven't paid: the Saga receives PAYMENT_FAILED → publishes compensating event → InventoryService releases reserved items → OrderService marks order CANCELLED → customer notified with "payment failed, try again." (3) During outage: optionally queue payment requests in Kafka with a retry processor — when Stripe recovers, process queued payments with idempotency keys (safe to retry). (4) Recovery: Circuit Breaker transitions to HALF_OPEN after 60 seconds, sends one test request, reopens on success. No manual intervention needed. (5) Operational: PagerDuty alert fires when circuit opens; SRE monitors Stripe status page + retry queue depth. This answer demonstrates: circuit breakers, Saga compensation, idempotency, event-driven recovery, and operational awareness.
